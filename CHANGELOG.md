## 2026-05-19 — Day 3 (in progress)

- **price-context-on-bets** — Closes the "wait, why did I lose?" gap. Three layers: (1) **`price_at_placement` column** added to `predictions` via migration `20260519000009`; `place_bet` RPC signature extended to accept `p_price_at_placement numeric default null` (nullable so a slow Massive lookup doesn't block placement). (2) **Live price captured at placement**: new `fetchLivePrice(ticker)` in `src/lib/price-history.ts` hits Massive's `/v3/snapshot/locale/us/markets/stocks/tickers/{ticker}` with a 1.5s AbortSignal cap, returns null on any failure. `placeBet` server action wires it into the RPC call; bet still ships with NULL if the fetch fails. (3) **Price action surfaced in UI**: BetHistoryList rows show "open $X → close $Y · ±Z%" subline on resolved bets (colored by direction of price movement, not outcome — a +1% move is green regardless of whether the user won or lost). Result reveal modal shows a new "Your entry · $X" row above the open/close pair, so the user sees their actual placement price alongside the resolution prices. Pure transparency add — resolution math unchanged, still open→close per ADR 0008.
- **delayed-bet-ui** — Status-aware UI for bets that fell through the cron's cracks. New `isStuckPrediction(bet, todayEt)` helper in `src/lib/bets.ts` derives a "stuck" state when `prediction_date < today_et AND !resolved` — covers cron failures, Yahoo price-data hiccups, weekend bets, and any other reason a resolution didn't fire on schedule. Wired into three surfaces: (1) `/bets` history page shows an amber "N bets are awaiting resolution · your stake is safe" banner when any rows match, plus a stronger amber "Delayed" badge variant (vs the normal "Pending" tone) on each delayed row. (2) BetCta's locked-in chip on home + detail pages swaps the direction tone for amber and replaces "Resolves in 3h" copy with "Resolution delayed" + AlertCircle icon. (3) Today's ET date threaded from server pages (`etCalendarDate()`) through StockCard → BetCta to keep the derivation deterministic. No schema change, no cron — pure UI derivation makes the situation visible rather than silently stuck. Pairs naturally with a future auto-VOID sweeper but doesn't require it.
- **local-finbert-and-hf-breaker** — Pipeline reliability rewrite, motivated by a 45-min `fetch_insights` timeout on free HF tier (logs showed ~500 FinBERT `ReadTimeoutError`s from cold-start latency + dozens of summarizer/reasoner `429 Too Many Requests` from the Inference Providers router's shared per-provider burst cap). Two changes ship together: (1) **FinBERT moved to local execution** — `pipeline/processors/sentiment.py` rewritten to load the ProsusAI/finbert model via `transformers` + CPU-only `torch` and score all of a stock's articles in a single batched forward pass. Lazy model load on first call; ~3 min cold download, sub-second per article warm. `requirements.txt` adds `transformers>=4.40` + `torch>=2.2`; `fetch-insights.yml` installs with `--extra-index-url https://download.pytorch.org/whl/cpu` (CPU wheel is ~200 MB vs CUDA's 800 MB) and adds an `actions/cache@v4` step keyed on `sentiment.py` + `requirements.txt` so `~/.cache/huggingface` persists across runs. Workflow `timeout-minutes` bumped 45 → 60 for headroom on first cold-cache fill. (2) **Shared HF circuit breaker for the remaining LLM calls** — new `pipeline/processors/_hf_breaker.py` tracks consecutive failures across both `LlamaSummarizer` and `VerdictReasoner`. After `TRIP_THRESHOLD=5` consecutive failures, subsequent HF calls short-circuit immediately to the rule-based fallback (verdict reasoning) or simply skip writing the field (summaries) — saves 90s × remaining-calls when HF is clearly down. Self-healing: a single successful call resets the counter, so a transient wobble doesn't permanently disable the path. Final `pipeline_done` log line now includes `hf_tripped=true/false hf_skipped=N` for one-line post-mortem. Numerical signal + verdict direction are never blocked by an HF outage. About page + README updated to reflect the split (FinBERT local, LLM still on HF + breaker). See [ADR 0012](docs/adr/0012-local-finbert-and-hf-breaker.md).
- **pull-to-refresh** — Native-feeling pull-to-refresh on the home feed. Custom hook over a library (~80 LOC, no new deps) — `PullToRefresh` client wrapper listens for pointer events globally, gates on `pointerType === "touch"` so desktop mice/trackpads are completely inert, only arms when `scrollY === 0`. Damping factor of 2.5 matches iOS native feel; user pulls 175px to hit the 70px refresh threshold. Single haptic tick when the threshold is crossed (and again if re-crossed after backing off), success haptic + spinner when released past threshold, then `router.refresh()` re-fetches the RSC tree. Indicator is a fixed-position pill that descends from the top with the pull amount, with the icon rotating proportionally before snapping to spin during refresh. Below-threshold release snaps back without doing anything. No keyboard equivalent intentionally — Next.js auto-refreshes via `revalidatePath` after mutations and users can always reload, so this is purely additive mobile polish.
- **haptics** — Tiny `haptic(pattern)` helper in `src/lib/haptics.ts` wraps `navigator.vibrate` with four named patterns (`tap`, `success`, `warning`, `double`). No-ops on SSR, on browsers without Vibration API (iOS Safari, desktop), and when the user has `prefers-reduced-motion: reduce` set — three layers of guardrail since this is non-critical polish. Wired into: BetSheet (tap on place success, warning on place error, double on cancel success, warning on cancel error), DailyBonusCard (success on streak claim, tap on day-1 claim, warning on error), ResultRevealModal (success on WIN reveal alongside confetti, tap on LOSS/VOID so the flip lands physically too). Android users get the feel; iOS + desktop see no change. Tuned to be felt-not-annoying: 12ms tap, 12-40-24 success pulse, 20-60-20-60-20 warning triple.
- **empty-states** — Shared `EmptyState` component centralizes the half-dozen ad-hoc empty cards across the app so they stay visually consistent and can iterate in one place. Server-component-safe (no client hooks); accepts a Lucide icon or custom ReactNode, title, description, optional internal-link CTA, and a `muted` / `card` variant. Retrofitted the empty states on /bets (no bets → Sparkles icon + "Browse today's signals" CTA back to home), /bets credit ledger (no activity → Coins icon), /leaderboard (no qualifiers → Trophy icon), and /stocks (no matching stocks → SearchX icon with query-specific copy). Badge grid on /profile keeps its guided locked-tile state since each locked tile already describes how to earn it.
- **upstash-ratelimit** — Wires Upstash Redis for the job it's actually meant for: per-user mutation rate limiting. README claimed Upstash was in the stack but no code was using it; this closes the gap. New `src/lib/rate-limit.ts` builds a lazily-initialized sliding-window `Ratelimit` per named action, keyed by `auth.uid()`. Applied to: `placeBet` 10/min, `cancelBet` 10/min, `claimDailyBonus` 3/min, `addToWatchlist` + `removeFromWatchlist` 20/min (shared limiter so a user can't dodge by alternating add/remove). All thresholds are upper bounds for honest users — anyone hitting them is either misbehaving or there's a runaway client. **Fails open**: when `UPSTASH_REDIS_REST_URL` / `UPSTASH_REDIS_REST_TOKEN` are unset (local dev) the limiter no-ops with a one-time warning, so the app stays functional. Also fails open on Upstash outages — a network blip shouldn't lock everyone out (logged for Sentry to pick up). When blocked, the action returns `error: "Slow down — try again in Ns"` with the actual retry-after computed from the limiter's `reset` timestamp.
- **docs-project-pitch** — README and `/about` rewritten around the "nightly research analyst" framing that makes the system legible to a first-time reader. README's *What it does* section now walks through the producer/consumer split (Python pipeline writes to Supabase → Next.js reads), what the 4 bucket scores actually capture and where the data comes from (Yahoo / Massive / Finnhub / SEC EDGAR / StockTwits / ApeWisdom / Reddit / FRED), why we use FinBERT and Llama-3 (context-sensitive sentiment + human-readable verdict reasoning, with graceful degradation when HF hiccups), and how the public track record is constructed. New *Where the numbers come from* section on `/about` mirrors the same narrative in four chunks — 90-second version, what gets pulled per stock, what the two AI models do, and a pointer to `pipeline/` for the audit trail — placed right before the existing four-bucket detail section so the rest of the page reads as a drill-down. No code changes, no schema impact.
- **browse-all-stocks** — New `/stocks` route closes the discovery gap: once onboarded, users could only ever see their 15 watchlist stocks; there was no way to explore the remaining 35 in the pool. Page lists all active stocks with client-side search (ticker + name fuzzy match) + sector filter pills + per-row Add/Remove toggle. Granular server actions `addToWatchlist(stockId)` / `removeFromWatchlist(stockId)` in `src/app/actions/watchlist.ts` enforce the 15-stock cap server-side, idempotent on unique-violation (re-adding a present stock is a silent no-op). Optimistic UI flips the toggle locally then unwinds with toast if the server rejects. Watchlist counter at top warns when cap is reached. revalidatePath wired so the home feed picks up adds/removes immediately. "Browse stocks" link added to ProfileMenu.
- **sentry-frontend** — `@sentry/nextjs@10` wired across all three Next runtimes (Node, edge, browser). `instrumentation.ts` is the entry point per Next 13.4+ conventions — branches on `NEXT_RUNTIME` to load `sentry.server.config.ts` or `sentry.edge.config.ts` lazily so we don't pull Node-only SDK into edge bundles. `instrumentation-client.ts` handles the browser SDK (replaces the old `sentry.client.config.ts` pattern in SDK 8+); enables session replay sampled only on error (0% clean / 100% errored), masks text + media in replays (PII guardrail), filters known-noise errors (ResizeObserver loops, AbortError, cross-origin script errors), denies extension URLs. `next.config.ts` wrapped with `withSentryConfig` — uploads + deletes source maps post-upload so the unminified maps stay private but stack traces symbolicate. Releases auto-tagged from `VERCEL_GIT_COMMIT_SHA`; env tagged from `VERCEL_ENV`. Tracing: 10% in prod, 100% in dev. New `SentryUserIdentifier` client component mounted in root layout subscribes to Supabase auth state and calls `Sentry.setUser({id, email})` on sign-in, clears on sign-out — every event gets user context. Whole thing **no-ops cleanly when `NEXT_PUBLIC_SENTRY_DSN` is unset**, so local dev without a DSN works fine. SETUP updated.
- **deploy-prep** — Vercel deployment prepped end-to-end. New `vercel.json` pins region (iad1) + adds CDN cache headers for the `/og/*` routes. New `docs/DEPLOYMENT.md` walks through: project creation, full env-var matrix (which go where, which are public vs server-only), Supabase auth redirect URL config for production + preview + localhost, Google OAuth redirect URI requirements, 10-step post-deploy smoke checklist, custom-domain wiring, and rollback procedure. Clarifies the split: Next.js on Vercel, Python pipeline + cron jobs on GitHub Actions (per ADR 0004). README + SETUP linked to the new doc; the stale Vercel section in SETUP redirects there as the single source of truth.
- **leaderboard-v1** — Weekly accuracy leaderboard. New `pipeline/compute_leaderboard.py` cron job runs Sundays 23:00 UTC via `.github/workflows/compute-leaderboard.yml` — pulls all predictions resolved Mon-Sun of the just-completed week, groups by user, filters to ≥5 decisive bets (WIN+LOSS, VOIDs excluded from denominator since flat-tape isn't a wrong call), ranks by accuracy desc then decisive count as tiebreaker, computes net `credits_won = sum(payout − wager)`, assigns tier (rank 1 → diamond, 2-3 → platinum, 4-10 → gold, 11+ → null). Idempotent — clears the week's rows then inserts fresh, so `workflow_dispatch` re-runs upsert cleanly. New `weekly_leaderboard_snapshots.display_name` column denormalized at snapshot time — required because the leaderboard read path runs in the user's auth-context which can't read other users' `user_profiles` (own_read RLS). New `/leaderboard` route: weekly heading + qualifier list with crown/medal icons for top 3 in tier colors, current user's row subtly highlighted. "Leaderboard" link added to ProfileMenu. Empty state: "First qualifier needs 5+ resolved bets in the week." Migration: `20260519000008_leaderboard_display_name.sql`.
- **badges-v1** — Six-badge catalog ships with the right detection point for each. **First Bet** awarded inside `place_bet` RPC (cheap detector: `total_predictions = 1` after the txn-local increment). **First Win** awarded by the resolution job — after the per-prediction `user_profiles` update, if `new_correct == 1 AND outcome == WIN`, call `award_badge` RPC (service-role only, idempotent via `ON CONFLICT DO NOTHING`). **Streak 3 / 7 / 14 / 30** awarded inside `claim_daily_bonus` when the new streak hits each threshold exactly (since reaching N requires having been at N-1 the day before, each threshold can only fire once per user per lifetime). New `src/lib/badges.ts` is the single source of truth for catalog (label / description / emoji / tier). New `/profile` route shows display name + member-since + 4-stat grid (predictions, accuracy, current streak, longest streak) + a 2x3 badge grid where earned tiles glow at their tier color (bronze / silver / gold / platinum) and unearned tiles are greyed with a lock + how-to-earn hint. "Profile & badges" link added to ProfileMenu. Migration: `20260519000007_badges.sql`.
- **signal-quality-p0-fixes** — Three structural fixes to the signal engine, motivated by a hedge-fund-analyst-style review of the model: (1) **MarketMind verdict resolution moved to `prev_close → close`** — the verdict is frozen at 8 PM ET T-1, so scoring it against `open → close` discarded the overnight gap (where most pre-open news is priced) and measured a different prediction than the one the model made. New `_fetch_mm_prices(ticker, target_date)` in `pipeline/resolve_predictions.py` pulls a 10-day window and returns `(prev_close, open, close)`; `_resolve_marketmind` now scores via `_outcome_against_reference(direction, reference_price=prev_close, close_price=close)`. User predictions still use `_evaluate()` against `open → close` per ADR 0008 — the bet-locking time matches the open, not prev_close. The `marketmind_predictions.open_price` column still stores the session's actual open (display fidelity preserved, no schema change); prev_close is in the resolution log line for now (`mm_resolve_scored ticker=X prev_close=Y close=Z outcome=W`) pending a future `reference_price` column. (2) **Defensive PIT filter on news** — new `_apply_pit_filter` in `pipeline/fetch_insights.py` drops articles whose `published_at` is more than 15 min in the future (publisher timestamp bugs) or older than 7 days (stale articles that previously leaked into top-3 display sorted by `|sentiment|` and into `cross_source_agreement` counts). Per-ticker drop counts logged. (3) **Verdict weights renormalize over present buckets** — `compute_verdict` no longer coerces `None → 0`, which had silently imposed a bearish drag whenever a positive bucket lacked corroborating data (a +0.4 technical with no professional read got diluted to 0.12 and fell below the 0.15 direction threshold). New behavior: missing buckets are excluded, remaining weights renormalized; all-None returns `NEUTRAL` with confidence 0. Also made `huggingface_hub` imports lazy inside `VerdictReasoner` so `compute_verdict` is testable without the ML deps. 12 unit tests in `pipeline/tests/test_verdict.py` cover renormalization, threshold boundaries, and the all-None safe default. See [ADR 0011](docs/adr/0011-signal-quality-p0-fixes.md).
- **result-reveal-modal** — Resolved bets now get a proper moment. New `revealed_at timestamptz` column on `predictions` + partial index `(user_id) WHERE resolved AND revealed_at IS NULL` for cheap lookup, paired with `mark_predictions_revealed(uuid[])` RPC (SECURITY DEFINER, scoped to `auth.uid()`, silently skips non-owner ids to avoid existence leaks). Home page parallel-fetches up to 10 unrevealed resolved bets via `fetchUnrevealedResolved`. New `ResultRevealModal` (framer-motion + canvas-confetti) auto-opens with a 3D card flip per bet — front face shows the bet, tap flips to reveal WIN/LOSS/VOID with payout + open/close prices. Confetti shower fires on each WIN, tuned to fall away naturally (80 particles, 70° spread, 35 velocity). Respects `prefers-reduced-motion` — falls back to instant fade, no flip, no confetti. Marks bets as revealed individually as the user advances (so a mid-flow close still persists progress), or flushes remaining IDs on dismiss. **Architectural note**: chose Postgres column over Redis or localStorage for this durable per-user state — see [ADR 0010](docs/adr/0010-postgres-over-redis-for-seen-state.md). Deps: `framer-motion@12`, `canvas-confetti@1.9`. Migration: `20260519000006_revealed_at.sql`.
- **daily-login-bonus** — The ritual hook. New `claim_daily_bonus(p_today_date)` Postgres RPC (SECURITY DEFINER, FOR UPDATE lock on `user_profiles`) atomically: validates "not already claimed today", computes new streak (continues if last claim was yesterday, else resets to 1), awards `min(100 + (streak-1)*20, 300)` credits, updates `current_streak` / `longest_streak` / `last_login_date`, appends a `type='DAILY_BONUS'` ledger row. Server action `claimDailyBonus` in `src/app/actions/bonus.ts` computes today's date via the new `etCalendarDate()` helper (matches the rest of the app's ET-aware semantics). Read helper `getDailyBonusStatus` mirrors the RPC's "available?" logic + projects what the next bonus would be — feeds `DailyBonusCard` (banner above the schedule bar) which renders bright emerald with claim button when available, muted with flame + "come back tomorrow" once claimed. Toast on success: "Day 3 streak! · +140 credits". Migration: `20260519000005_claim_daily_bonus_rpc.sql`.

New `fetchDailyBars` in `src/lib/price-history.ts` hits Massive's `/v2/aggs/ticker/{ticker}/range/1/day/{from}/{to}` REST endpoint server-side, cached via Next's `fetch` with `revalidate: 3600` (daily bars only mint at close, hourly is plenty). Hover tooltip shows per-day date + close. New section header shows the last close + % change over the window. Required `MASSIVE_API_KEY` in `.env.local` (not just `pipeline/.env`); SETUP updated.

## 2026-05-18 — Day 2

- **summarizer-fixed** — Llama-3-8B-Instruct was failing with opaque `status=?` HTTPErrors — the model is gated on HF and the free serverless tier doesn't reliably serve it anymore. Switched to: (1) `chat_completion` API (newer, OpenAI-compatible, routes cleanly), (2) `mistralai/Mistral-7B-Instruct-v0.3` as default (well-supported on free `hf-inference` provider), (3) explicit `provider="hf-inference"` to bypass "auto" routing that lands on paid providers with opaque errors, (4) much richer error logging (exception type + body extract) so future failures debug in one log line. Model is configurable via `HUGGINGFACE_SUMMARY_MODEL` env var.
- **article-summaries-v2** — Articles now carry three structured fields (tldr / summary / signal_influence) instead of just one. New migration `20260518000003_article_richer_summary.sql` adds `summary` and `signal_influence` columns to `insight_articles`. `LlamaSummarizer` rewritten with a single labeled prompt that returns all three fields, parsed via regex with defensive trimming; if parsing fails for any field, the others still land. Unit tests in `pipeline/tests/test_summarizer.py` cover the parser's tolerance for mixed case, quotes, trailing punctuation, and partial output. `NewsArticle` dataclass extended; orchestrator inserts the new fields.
- **backtest-harness** — `pipeline/backtest.py`: replays the technical bucket against `--months` of historical OHLCV (yfinance + curl_cffi), computes accuracy per score band (high bullish, low bullish, etc.), writes a CSV plus prints a stdout summary. No lookahead — each prediction uses only data available at end of evaluation day. Sentiment/professional/social backtests deferred (need historical news + filings — Week 2+).
- **resolve-predictions-workflow** — `.github/workflows/resolve-predictions.yml`: cron at 21:15 UTC Mon–Fri (~4:15 PM ET, 15-min cushion for Yahoo to publish the close). `workflow_dispatch` with optional `--date` and `--dry-run`. Same `production` environment, only needs Supabase secrets.
- **resolve-predictions-job** — `pipeline/resolve_predictions.py`: fetches unresolved predictions for the target date, fetches each ticker's day bar once via yfinance + curl_cffi, evaluates `direction` vs open→close direction. WIN = stake × 1.8, LOSS = 0, VOID = stake refund (on flat days or price unavailable). Updates `predictions`, appends to `credit_transactions`, bumps `user_profiles.total_predictions` + `correct_predictions`. Idempotent — only touches rows where `resolved=false`.
- **social-signals** — `fetchers/stocktwits.py` (public API, no auth, bullish/bearish ratio from messages), `fetchers/apewisdom.py` (one cached request loads top ~500 WSB-mentioned tickers, looked up per-stock), `fetchers/reddit.py` (PRAW-backed mention counts across r/wallstreetbets, r/stocks, r/investing; gracefully skipped if creds absent). Orchestrator's `_build_social` coalesces all three into the `SocialSnapshot` consumed by the aggregator — fills the previously-null `social_score` bucket.
- **llama-summarizer** — `processors/summarizer.py`: `LlamaSummarizer` uses `huggingface_hub.InferenceClient` to call `meta-llama/Meta-Llama-3-8B-Instruct`. Generates a 1-sentence TL;DR (max 140 chars) for the top 3 articles per stock — only the ones we surface in the UI, keeping inference cost low. Mutates `NewsArticle.tldr` in place.
