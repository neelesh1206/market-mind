## 2026-05-19 — Day 3 (in progress)

- **finbert-revision-pin** — `sentiment.py` was calling `AutoModel.from_pretrained("ProsusAI/finbert")` with no `revision=` argument, which defaults to `main` HEAD on HuggingFace Hub at the moment of cache miss. Silent reproducibility hazard: if Prosus republishes the model, our next cache invalidation would download new weights and our `marketmind_predictions` accuracy track record becomes apples-to-oranges across that boundary. Also a (small) supply-chain attack surface — a compromised `ProsusAI` HF org could push poisoned weights and we'd consume them on next cache miss. Fixed by pinning `MODEL_REVISION` to `4556d13015211d73dccd3fdd39d39232506f3e43` (current `main` HEAD as of 2026-05-20; the model itself hasn't been republished since 2023-05-23 — research-grade frozen weights from the original Prosus paper). Passed to both `AutoTokenizer.from_pretrained()` and `AutoModelForSequenceClassification.from_pretrained()` via the `revision=` kwarg. **No behavioral change today** — the pin matches what `main` already points to — but locks us against future silent drift. Upgrade procedure documented inline + in ADR 0012's 2026-05-20 amendment: bump SHA in a PR, run `fetch_insights --dry-run`, diff bucket scores against prior run, only then merge. Log line now includes `revision=4556d130` short prefix for grep-ability. Mistral summarizer model still relies on its versioned name (`Mistral-7B-Instruct-v0.3`) for now; formal revision pinning there is the next hygiene item.
- **prediction-feedback** — Closes the feedback loop on the verdict surface. Users can now thumbs-up / thumbs-down each MarketMind verdict directly on `/stock/[ticker]`, with optimistic UI and aggregate counts visible to everyone (including anon visitors, who see the count + a sign-in CTA but can't vote). Schema: new `prediction_feedback` table with `(user_id, marketmind_prediction_id)` uniqueness — re-voting overwrites cleanly. Two SECURITY DEFINER RPCs: `submit_prediction_feedback(p_prediction_id, p_helpful, p_comment)` (idempotent upsert, validates auth, friendly-errors verdict_missing) and `get_feedback_summary(p_prediction_id)` (public aggregate count so anon visitors can see "9 of 13 found helpful"). RLS: per-user vote rows are own-only; aggregate flows through the security-definer function. UI: new `PredictionFeedback` client component with three display modes based on sample size — `N=0` shows "Be the first to weigh in", `N<5` shows "X of Y people found it helpful" (no percentage to avoid 1/1=100% misleading display), `N≥5` shows the percentage. Server action `submitPredictionFeedback` wraps the RPC, rate-limited at 10/min/user (10/min added to the LIMITERS table in `rate-limit.ts` alongside placeBet/cancelBet/etc.). Optimistic vote-flip rolls back via state-restore + sonner toast on RPC failure. Wired into `/stock/[ticker]` between the per-stock track-record and the bucket breakdown — same section as the verdict chip. 8 Vitest cases on the lib helpers cover RPC-array-row mapping, defensive zero-fallback when migration is pending, anon-skips-query early return, vote-exists / vote-missing / query-error paths. Migration: `20260520000002_prediction_feedback.sql`.
- **verdict-honesty-pass** — Two UX-honesty improvements to the daily verdict surface. (1) **NEUTRAL chip visually distinct from UP/DOWN** — `VerdictChip` now renders NEUTRAL with a dashed-border, transparent-background chip with a `HelpCircle` icon and explicit "no clear read today" copy (no percentage shown). UP/DOWN keep their solid colored chips. The distinction matters because the previous design made NEUTRAL look like a third tone in the same palette, which read as "MarketMind is mildly indecisive" — actually it means "we deliberately didn't make a call." Companion copy on the reasoning row: when there's no LLM reasoning text for a NEUTRAL row, we now show "Signals are pointing in different directions — we don't have a confident read on this one today. Better to skip than to call a coin flip." (2) **Richer fallback reasoning text** — pipeline-side. `Verdict` dataclass gains an optional `breakdown` field (the same aggregator JSON shape that ships in `stock_insights.signal_breakdown`), populated in `fetch_insights.py` right after `compute_verdict()`. `_fallback_reasoning` rewritten with four per-bucket describe helpers (`_describe_professional`, `_describe_technical`, `_describe_sentiment`, `_describe_social`) that surface concrete fragments — e.g. *"Bullish — 12 of 14 analysts rate Buy, recent upgrade; oversold RSI, MACD bullish crossover."* instead of the previous generic *"Bullish — driven primarily by professional and technical signals."* NEUTRAL gets a richer enumeration too: *"Mixed — 10 of 14 analysts rate Buy pulling up, overbought RSI pulling down. No clear read."* Backwards compatible: callers without a breakdown attached fall through to the old name-based phrasing. 6 new tests in `pipeline/tests/test_verdict.py` (28 verdict tests total now) cover the with-breakdown / without-breakdown / bullish / bearish / NEUTRAL-enumeration / all-None branches. Not yet wired into the LLM prompt — the LLM still gets only score numbers; the fallback enhancement closes the gap when the LLM is unavailable, which is the harder problem to solve well.
- **entry-vs-close-resolution** — Fairness fix for the bet-window-during-market-hours design. ADR 0008 extended the bet window past market open (8 PM prev-day → 1 PM ET) but kept everyone on `open → close` resolution. That created a measurable inequity: a user betting UP at 12:30 PM with the stock already up 2% only needed the close to *stay* above the open, while someone betting UP the night before was held to the same bar despite having no information about the day's tape. Now that `price_at_placement` is captured on every bet (#125) and Finnhub real-time quotes back it accurately, we can fairly score in-market bets against the price the user actually saw. New two-mode resolver (ADR 0017): **bets created at or after 9:30 AM ET on the trading day, with a recorded entry price** resolve `entry → close`; **everything else** (pre-market, grandfathered, or in-market bets where Finnhub was down) stays on `open → close`. Existing bets fully grandfathered via a hardcoded `RESOLUTION_V2_CUTOFF` constant — users who placed under the old rules resolve under the old rules. Scoring extracted from `resolve_predictions.py` into new pure-math module `pipeline/processors/resolution_scoring.py` (mirrors the verdict.py separation pattern — tests don't need pandas/yfinance). 20 Python tests lock in every branch of the discriminator: grandfathered-takes-precedence, in-market-entry-wins, pre-market-uses-open, NULL-entry-falls-back, 0-entry-falls-back, exact-9:30-AM-treated-as-pre-market, DST boundaries (EDT vs EST giving correct UTC offsets), end-to-end "bet that would have won under open-mode now loses under entry-mode". UI side: `bets.ts` exports a mirrored `resolutionReferenceFor(bet)` helper used by `BetHistoryList` to show "entry $X → close $Y" vs "open $X → close $Y" with the matching reference label. Result reveal modal already displays all three (entry / open / close) so it didn't need changes. See [ADR 0017](docs/adr/0017-entry-vs-close-resolution-for-in-market-bets.md) for the full alternatives matrix (stake decay, tighter window, entry-mode for every bet, dedicated `resolution_mode` column — all rejected with rationale).
- **honest-accuracy-display** — Track-record now shows the **95% Wilson confidence interval** alongside the headline accuracy — so users can see at a glance how much to trust the number given the sample size we have. New `src/lib/wilson.ts` exports `wilsonInterval(correct, total, z)` returning `{lower, upper, center}`; uses standard Wilson score interval (no continuity correction), matching the Wikipedia formula and `statsmodels.proportion_confint(method='wilson')`. Why Wilson over the normal approximation: it stays bounded to [0, 1] even at extreme proportions, doesn't pull apart at p=0/p=1, and is the default in stats packages (Brown-Cai-DasGupta 2001 confirmed it as the small-sample best choice). 12 unit tests lock in known reference values, edge cases (n=0, p=0, p=1), and monotonic CI-shrinks-with-N. `fetchTrackRecord` returns `{total, correct, accuracy, ciLower, ciUpper}`; new `fetchStockTrackRecord(client, stockId)` does the same scoped to one stock. `TrackRecordBadge` displays the CI inline like *"Right 18 of 27 · 67% (48–81%)"* with a tooltip explaining the math — collapsed to just the headline via `compact={true}` for the stock-card footer. **New per-stock track record** surfaced on `/stock/[ticker]` under the verdict chip — *"On AAPL: Right 4 of 7 · 57% (25–84%)"* — so accountability is visible per name, not just universe-wide. **`/about` rewritten** as "How we're doing" with a 2×2 multi-window grid (7d / 30d / 90d / all-time), a sample-size-aware honesty caption above the grid that adapts copy based on resolved count (no data → "first day's data lands tomorrow", <30 → "wide on purpose", <100 → "still wide", ≥100 → "narrowed enough to be defensible"), and a collapsible explainer for what the CI actually means. **Feedback collection** added directly to the "How we're doing" section — GitHub issue link with a pre-filled `feedback` label + a mailto fallback, framed as "honest critique on whether the calls are helping you make sense of the market is the single most valuable input we can get right now." This is the chunk-1 of two from the post-analyst-review honesty work — chunk-2 (in-app thumbs UI + more prominent NEUTRAL chips + better fallback reasoning text) lands next.
- **live-prices-finnhub-swap** — Correction to the **live-prices-ui** entry below: the original implementation hit Polygon's `/v3/snapshot` endpoint based on the (incorrect) claim that their free tier delivers 15-min delayed quotes. Verified against `api.polygon.io` with our key — snapshots return **403 NOT_AUTHORIZED** on free tier, gated to the $29/mo Starter plan. The bug surfaced when the UI silently rendered `prev_close` everywhere (because `getLivePrices()` returned all-null entries). Swapped data source from Polygon snapshots to **Finnhub's `/quote` endpoint** — free-tier, 60 calls/min, 30/sec, **real-time** (not delayed) US equity quotes. Response shape mapped: `c` (current) → `price`, `dp` (change %) → `changePct`, with `c=0` treated as "unknown ticker" sentinel that surfaces null. Env var `MASSIVE_API_KEY` → `FINNHUB_API_KEY` (key already existed as a GH Actions secret for the pipeline's analyst-consensus fetcher; added to Vercel env separately). Everything else carries over unchanged — same Upstash cache layer, same 5min positive / 1min negative TTLs, same graceful-failure path. UI copy updated from "Live · ~15 min delayed" to just "Live · Finnhub" since quotes are now real-time. 13 Vitest cases rewritten to mock the Finnhub response shape; added a regression test ensuring the API key never leaks into warn logs. **Engineering lesson logged** — when an API tier is load-bearing for a design decision, verify with one curl call before shipping. The marketing copy on a pricing page is not a substitute for hitting the endpoint.
- **computed-at-refreshes-on-upsert** — Fix a misleading "updated N hours ago" label on stock cards. `stock_insights.computed_at` was defined as `timestamptz not null default now()`, but Postgres only applies `default` on `INSERT` — when the upsert hits the conflict path and `UPDATE`s an existing row, the original `computed_at` is preserved. Verified via `stock_insight_sources` audit log: AMZN's May-20 row was touched by 5 separate runs (May 18 7:54 PM → May 19 2:34 PM PDT) but the UI kept showing the original-insert timestamp because subsequent upserts didn't refresh it. One-line fix: `_build_insight_payload` now sets `"computed_at": datetime.now(timezone.utc).isoformat()` explicitly in the upsert dict. Next run writes a fresh timestamp on every row touched; the UI's "updated N ago" label becomes honest. No schema change, no migration needed. Pre-existing rows stay stale until they're next overwritten (a future re-run for that date — or a one-off backfill if we ever want to repair the history; not worth doing for ephemeral display).
- **live-prices-ui** — Adds (15-min delayed) live stock prices to the home feed and stock detail page, backed by a shared Upstash Redis cache so all Vercel function instances see the same snapshot. New `src/lib/live-prices.ts` exposes `getLivePrices(tickers[])` and `getLivePrice(ticker)` that: (1) MGET all tickers from `mm:price:<TICKER>` keys in one Redis round-trip, (2) fetch Polygon's `/v3/snapshot/locale/us/markets/stocks/tickers/{T}` endpoint in parallel via `Promise.allSettled` for misses (so one bad ticker can't break the page), (3) write fresh values back with **5min TTL for successes**, **1min for nulls** (negative cache stops us hammering a failing ticker without committing a 5-min black mark). Why shared Redis instead of Next's `unstable_cache`: per-instance caches under Vercel's serverless model would multiply Polygon calls by instance count, blowing the free-tier 5-calls/min quota the moment traffic picks up; shared Upstash gives us O(stocks) calls per cache window regardless of user count. **Graceful degradation everywhere** — no Upstash creds → direct Polygon (works, slow); no Polygon key → all nulls (UI shows "—"); Polygon 4xx/5xx/timeout → null entry (UI shows "—" instead of crashing); Redis network blip → fall through to direct Polygon. `StockCard` gains an optional `livePrice` prop; when present, the header shows current price + today's % change with a "· live" microlabel; falls back to the pipeline's `prev_close` + weekly change otherwise. Stock detail page header similarly swaps the `day` change column for the live one when available and prints a "Live · ~15 min delayed" disclaimer. Ticker normalization mirrors the Python pipeline (BRK.B → BRK-B on the outbound URL, dotted form preserved on the returned Map key for DB joins). 13 new Vitest cases lock in the parser branches (lastTrade vs min.c vs day.c fallback chain, todaysChangePerc vs derived-from-prevDay, null on missing fields), the failure modes (4xx/5xx, abort, missing API key, fetch throws), dedup on repeated tickers, and dotted-ticker normalization. Vitest config also got a `workers/**` exclude to keep vitest from accidentally running cron-trigger's transitive-dep test files.
- **external-cron-cloudflare-worker** — Replaces GitHub Actions `schedule:` triggers with a Cloudflare Worker. GH Actions' scheduled triggers are best-effort and we observed ~3hr drift on the first eligible `fetch-insights` run (target 17:00 PDT, fired 19:53 PDT); GH's own docs warn schedules can be delayed *hours* during high load and occasionally skipped entirely. For a daily-ritual app where a missed pipeline = a broken day, best-effort isn't acceptable. New `workers/cron-trigger/` package (~80 LOC TS Worker + wrangler.toml + README + tsconfig + package.json) declares three CF cron triggers — `0 0 * * 2-6` (fetch-insights), `15 21 * * 1-5` (resolve-predictions), `0 23 * * 0` (compute-leaderboard) — and on each fire looks up the matching workflow file in a `CRON_TO_WORKFLOW` map and POSTs to GitHub's `/actions/workflows/{file}/dispatches` API with a fine-grained PAT. CF Workers cron is reliable within ~1 min vs GH's multi-hour drift. **`schedule:` blocks removed from all three workflows** — they now only expose `workflow_dispatch:`, and the Worker is the single source of truth for pipeline timing. Each workflow's existing `concurrency: { cancel-in-progress: false }` block prevents duplicate-run risk from a manual + cron collision. Cost: $0 (Workers Free tier, ~90 invocations/month vs 100k/day quota). New vendor footprint: zero — user already runs Cloudflare for the custom-domain DNS. Worker deploy is `npx wrangler deploy`; PAT stored via `wrangler secret put GITHUB_PAT`. Failure mode: dispatch errors throw and surface in `wrangler tail` + CF dashboard logs; no automatic retry, but pipeline health view (#122) will surface missed runs to users when shipped. See [ADR 0016](docs/adr/0016-external-cron-via-cloudflare-worker.md) for the alternatives matrix (cron-job.org, self-hosted cron, AWS EventBridge) and [workers/cron-trigger/README.md](workers/cron-trigger/README.md) for one-time setup steps.

- **vol-normalize-direction-threshold** — Per-stock signal-to-noise calibration in the verdict layer. The old `DIRECTION_THRESHOLD = 0.15` was applied flat to every ticker, which under-called quiet names like PG (daily σ ≈ 0.9%) and over-called noisy ones like NVDA (σ ≈ 3.5%) — same magnitude of combined score is much more meaningful when the underlying name barely moves. `PriceSnapshot` gains a `realized_vol_20d: float | None` field, computed in `YFinancePriceFetcher._build_snapshot` as stddev of the last 20 daily returns (no extra network call — same dataframe). `compute_verdict` accepts it as an optional kwarg and scales the threshold: `vol_factor = clamp(realized_vol/0.02, 0.5, 2.5)`, then `adjusted_threshold = 0.15 * vol_factor`. So NVDA needs `|combined| > 0.26` to flip directional; PG flips at `> 0.075`. The `Verdict` dataclass carries `adjusted_threshold` + `vol_factor` for audit; the per-stock log line in `fetch_insights.py` now emits `vol=0.0234 factor=1.17 threshold=0.176`. Callers that omit `realized_vol_20d` get the prior flat-0.15 behavior — no regression. 6 new tests in `pipeline/tests/test_verdict.py` cover the high-vol-needs-stronger-signal, low-vol-flips-on-modest-signal, clamp-at-extremes, defensive-on-bad-vol, and renormalization-composes invariants. Why fixed-prior reference vol instead of universe-median: universe changes shouldn't shift every other stock's threshold; market-regime shifts belong in a separate regime layer, not embedded here. See [ADR 0014](docs/adr/0014-vol-normalize-direction-threshold.md).
- **social-bucket-fade-the-crowd** — Fix a sign-error in the social bucket flagged by the analyst review: the old `social_score` in `pipeline/processors/aggregator.py` treated every positive social signal as bullish (mention spikes, top-WSB rank, StockTwits bullish ratio all added with `+` weight), but the academic literature (Barber & Odean 2008; Da, Engelberg & Gao 2011; subsequent work in the cross-section) is consistent that **retail-attention spikes precede underperformance** for non-meme tickers — the crowd shows up after news is priced, latecomers buy the top. Rewritten to split the bucket into two explicit components: (1) **Herding** (Reddit mention delta + ApeWisdom rank) contributes **negatively** with magnitude scaling by intensity (delta >500% → -0.4, ApeWisdom top-3 → -0.3, etc.), and produces a 0..1 `herding_intensity` value that gates the second component. (2) **Informed sentiment** (StockTwits bullish ratio) stays directional but is **volume-damped** (drops to 0.3× weight when message count >2000 — viral threshold) AND **herding-damped** (multiplied by `1 − 0.7 × herding_intensity`, so peak herding cuts the directional weight to 0.3×). The bucket no longer *inverts* the StockTwits sign at peak herding — that would require stronger empirical justification than current literature supports — it damps toward zero, expressing reduced conviction rather than active contrarianism. New `breakdown.social` JSON exposes `herding_intensity`, `herding_damping`, and `stocktwits_volume_weight` for audit. 7 unit tests in `pipeline/tests/test_social_score.py` lock in the meme-flips-sign / quiet-mega-cap-no-regression / informed-low-volume / damping-monotonic invariants. About page Social section rewritten to describe the new behavior. Old `marketmind_predictions` rows aren't strictly comparable to post-change rows; tracking continues via the existing `weights_version` column. See [ADR 0013](docs/adr/0013-social-bucket-fade-the-crowd.md).
